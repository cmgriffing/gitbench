# Benchmark execution orchestration lives in `BenchmarkRunner`, not CLI

The `run_benchmark` and `run_model_benchmarks` functions (fixture lifecycle, parallelism, progress reporting) lived in `cli.py` alongside Click commands, TTY tables, and ANSI codes. We extracted them into `gitbench/harness/runner.py` as `BenchmarkRunner`, which takes a benchmark registry and a model client at construction time and exposes `run_benchmark()` and `run_all()`. The `RunProgress` protocol moved with it. The CLI is now responsible only for option parsing, wiring components together, and display output — it no longer manages `ThreadPoolExecutor` or `executor.cleanup()` internals. Per-fixture verbose output was temporarily lost and should be restored via `RunProgress` when `TerminalProgressTable` is extracted.
